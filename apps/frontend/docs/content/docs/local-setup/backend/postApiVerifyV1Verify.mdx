# Verify Single Email (Authenticated)

Verify email addresses using your API key with automatic credit management and persistent storage of results in your Verifio dashboard.

## What This API Does

This endpoint verifies a single email address **with authentication**, which means:
- ‚úÖ Results are **saved to your dashboard** for future reference
- ‚úÖ **Credits are automatically deducted** from your account
- ‚úÖ **No rate limits** (unlike the public endpoint)
- ‚úÖ Full **verification history** and analytics tracking
- ‚úÖ Access to **advanced features** and priority processing

## When to Use This API

| Use This Endpoint | Use Public Endpoint Instead |
|-------------------|----------------------------|
| Production applications | Quick testing without signup |
| Need to track verification history | One-off verifications |
| Building user-facing features | Prototyping |
| Require credit-based billing | Don't want to manage credits |
| Need higher rate limits | Occasional use (2/min is enough) |

<Callout type="info">
  **Key Difference:** This endpoint requires an API key and deducts credits, but saves results to your dashboard and has no rate limits. The public endpoint (`/v1/email`) is rate-limited but free for testing.
</Callout>

### In Your Verifio Dashboard
When you verify emails through the Verifio web interface, this is the API being called behind the scenes. Your verification history page shows all results from this endpoint.

### Common Integration Points

- **CRM Systems:** Clean and verify contact lists

```go
package main

import (
    "bytes"
    "encoding/json"
    "fmt"
    "io"
    "net/http"
    "os"
)

type VerifyRequest struct {
    Email string `json:"email"`
}

type VerifyResponse struct {
    Success   bool   `json:"success"`
    Data      Result `json:"data,omitempty"`
    Error     string `json:"error,omitempty"`
    RequestID string `json:"requestId"`
}

type Result struct {
    ID    string `json:"id"`
    Email string `json:"email"`
    State string `json:"state"`
    Score int    `json:"score"`
}

func main() {
    apiKey := os.Getenv("VERIFIO_API_KEY")
    
    reqBody, _ := json.Marshal(VerifyRequest{
        Email: "user@example.com",
    })
    
    req, _ := http.NewRequest(
        "POST",
        "https://verifio.email/api/verify/v1/verify",
        bytes.NewBuffer(reqBody),
    )
    
    req.Header.Set("Authorization", "Bearer "+apiKey)
    req.Header.Set("Content-Type", "application/json")
    
    client := &http.Client{}
    resp, err := client.Do(req)
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    defer resp.Body.Close()
    
    body, _ := io.ReadAll(resp.Body)
    
    var result VerifyResponse
    json.Unmarshal(body, &result)
    
    if result.Success {
        fmt.Printf("ID: %s\n", result.Data.ID)
        fmt.Printf("State: %s\n", result.Data.State)
        fmt.Printf("Score: %d\n", result.Data.Score)
    } else {
        fmt.Printf("Error: %s\n", result.Error)
    }
}
```

## Request Format

### Headers

| Header | Required | Value | Description |
|--------|----------|-------|-------------|
| `Authorization` | ‚úÖ Yes | `Bearer YOUR_API_KEY` | Your Verifio API key |
| `Content-Type` | ‚úÖ Yes | `application/json` | Must be JSON |

### Request Body

```json
{
  "email": "user@example.com",
  "options": {
    "skipDisposable": false,
    "skipRole": false,
    "skipTypo": false
  }
}
```

#### Parameters

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `email` | string | ‚úÖ Yes | Email address to verify (1-254 characters, valid email format) |
| `options` | object | ‚ùå No | Verification options to customize behavior |
| `options.skipDisposable` | boolean | ‚ùå No | Skip disposable email detection (default: `false`) |
| `options.skipRole` | boolean | ‚ùå No | Skip role-based email detection (default: `false`) |
| `options.skipTypo` | boolean | ‚ùå No | Skip typo suggestion (default: `false`) |

<Callout>
  **Options Usage:** Only skip checks if you have specific requirements. For example, skip `skipDisposable` if you explicitly allow temporary emails. Skipping checks speeds up verification slightly but provides less information.
</Callout>

### Response Fields

#### Top Level

| Field | Type | Description |
|-------|------|-------------|
| `success` | boolean | `true` if verification succeeded, `false` if error |
| `data` | object | Verification result (only present when `success` is `true`) |
| `error` | string | Error message (only present when `success` is `false`) |
| `requestId` | string | Unique request identifier for debugging |

#### Data Object

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | **Unique verification ID** - use this to retrieve the result from your dashboard |
| `email` | string | The email address that was verified |
| `user` | string | Local part (before @) |
| `domain` | string | Domain part (after @) |
| `tag` | string\|null | Plus-addressing tag (e.g., `newsletter` in `user+newsletter@example.com`) |
| `state` | string | Verdict: `deliverable`, `undeliverable`, `risky`, or `unknown` |
| `reason` | string | Specific reason code explaining the verdict |
| `score` | number | Quality score (0-100, higher is better) |
| `checks` | object | Detailed results from each verification check |
| `analytics` | object | Advanced insights about email quality and risk |
| `duration` | number | Verification time in milliseconds |
| `verifiedAt` | string | ISO 8601 timestamp of when verification completed |

### State Values

| State | Meaning | Recommended Action |
|-------|---------|-------------------|
| `deliverable` | ‚úÖ Email is valid and will receive messages | Safe to send emails |
| `undeliverable` | ‚ùå Email is invalid or doesn't exist | Block the email, show error to user |
| `risky` | ‚ö†Ô∏è Email exists but has quality concerns | Use caution, may bounce or have low engagement |
| `unknown` | ‚ùì Could not determine validity | Retry later or accept with manual review |

### Score Interpretation

| Score | Quality | Recommendation |
|-------|---------|----------------|
| 90-100 | Excellent | ‚úÖ High deliverability, safe to send |
| 70-89 | Good | ‚úÖ Generally safe, minor concerns possible |
| 50-69 | Fair | ‚ö†Ô∏è Some risk factors, monitor bounce rates |
| 30-49 | Poor | ‚ö†Ô∏è High risk, consider not sending |
| 0-29 | Very Poor | ‚ùå Very high bounce risk, do not send |

## Error Handling

### HTTP Status Codes

| Code | Meaning | What Happened | What to Do |
|------|---------|---------------|------------|
| `200` | Success | Verification completed | Parse the response |
| `400` | Bad Request | Invalid email format or request body | Check your request format |
| `401` | Unauthorized | Invalid or missing API key | Verify your API key is correct |
| `402` | Payment Required | Insufficient credits | Add credits to your account |
| `429` | Too Many Requests | Rate limit exceeded (rare with auth) | Slow down requests |
| `500` | Server Error | Internal error occurred | Retry after a moment |
| `503` | Service Unavailable | Upstream service failure | Retry with exponential backoff |

### Error Response Format

```json
{
  "success": false,
  "error": "Insufficient credits",
  "requestId": "req_xyz789abc123",
  "data": {
    "remaining": 0,
    "required": 1
  }
}
```

### Common Errors

#### 1. Insufficient Credits (402)

```json
{
  "success": false,
  "error": "Insufficient credits",
  "requestId": "req_abc123",
  "data": {
    "remaining": 0,
    "required": 1
  }
}
```

**What to do:**
- Add credits to your account via the dashboard
- Implement credit monitoring in your application
- Set up low-credit alerts

#### 2. Invalid API Key (401)

```json
{
  "success": false,
  "error": "Invalid API key",
  "requestId": "req_abc123"
}
```

**What to do:**
- Verify your API key is correct
- Check if the key has been revoked or expired
- Ensure you're using `Bearer` prefix in Authorization header

#### 3. Invalid Email Format (400)

```json
{
  "success": false,
  "error": "Validation error: email must be a valid email address",
  "requestId": "req_abc123"
}
```

**What to do:**
- Validate email format on client-side before sending
- Check for common mistakes (missing @, spaces, etc.)

#### 4. Verification Timeout (500)

```json
{
  "success": false,
  "error": "Verification timeout - request took too long",
  "requestId": "req_abc123"
}
```

**What to do:**
- Retry the request (SMTP server may have been slow)
- If persistent, the domain's mail server may be unresponsive
- Consider marking as `unknown` and allowing with manual review

```javascript
async function verifyWithRetry(email, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      const result = await verifio.verify(email);
      return result;
      
    } catch (error) {
      // Don't retry on client errors
      if (error.status === 400 || error.status === 401 || error.status === 402) {
        throw error;
      }
      
      // Retry on server errors or timeouts
      if (attempt < maxRetries) {
        const waitTime = Math.pow(2, attempt) * 1000; // Exponential backoff
        console.log(`Attempt ${attempt} failed. Retrying in ${waitTime}ms...`);
        await new Promise(resolve => setTimeout(resolve, waitTime));
      } else {
        throw error;
      }
    }
  }
}

// Usage
try {
  const result = await verifyWithRetry('user@example.com');
  console.log('Verification successful:', result.state);
} catch (error) {
  if (error.status === 402) {
    console.error('Out of credits! Please add more.');
  } else if (error.status === 401) {
    console.error('Invalid API key');
  } else {
    console.error('Verification failed after retries:', error.message);
  }
}
```

## Best Practices

### 1. Store API Keys Securely

**‚ùå Never do this:**
```javascript
const apiKey = "vrf_sk_live_abc123xyz789"; // Hardcoded!
```

**‚úÖ Do this instead:**
```javascript
// Use environment variables
const apiKey = process.env.VERIFIO_API_KEY;

// Or use a secrets manager
const apiKey = await secretsManager.getSecret('verifio-api-key');
```

### 2. Handle All States

Don't just check for `deliverable` - handle all four states:

```javascript
switch (result.state) {
  case 'deliverable':
    // Proceed normally
    break;
  case 'undeliverable':
    // Block and show error
    break;
  case 'risky':
    // Decide based on score and warnings
    break;
  case 'unknown':
    // Have a policy for unknowns
    break;
}
```

### 3. Use Typo Suggestions

Help users fix typos:

```javascript
if (result.checks.typo.hasTypo) {
  const suggestion = result.checks.typo.suggestion;
  // Show: "Did you mean user@gmail.com instead of user@gmial.com?"
}
```

### 4. Monitor Your Credits

Don't let your application run out of credits:

```javascript
// Check credits before bulk operations
const creditsNeeded = emailList.length;
const creditsAvailable = await verifio.getCredits();

if (creditsAvailable < creditsNeeded) {
  throw new Error(`Need ${creditsNeeded} credits, only have ${creditsAvailable}`);
}
```

### 5. Implement Retry Logic

Always retry on server errors (500, 503) but not on client errors (400, 401, 402):

```javascript
const shouldRetry = (error) => {
  return error.status >= 500 || error.code === 'TIMEOUT';
};
```

### 6. Cache Results

Don't verify the same email multiple times:

```javascript
const cache = new Map();

async function verifyEmailCached(email) {
  // Check cache first
  if (cache.has(email)) {
    return cache.get(email);
  }
  
  // Verify and cache
  const result = await verifio.verify(email);
  cache.set(email, result);
  
  return result;
}
```

### 7. Use the Verification ID

Save the `id` field to retrieve results later:

```javascript
// Save to your database
await db.users.update(userId, {
  verificationId: result.id,
  emailScore: result.score,
  emailState: result.state
});

// Later, you can look it up in your Verifio dashboard
// using this ID
```

## Common Mistakes to Avoid

### ‚ùå Mistake 1: Not Checking Credits Before Bulk Operations

```javascript
// BAD: Will fail halfway through
for (const email of largeEmailList) {
  await verifio.verify(email); // Might run out of credits
}
```

```javascript
// GOOD: Check credits first
const creditsNeeded = largeEmailList.length;
const creditsAvailable = await verifio.getCredits();

if (creditsAvailable < creditsNeeded) {
  throw new Error('Insufficient credits for bulk operation');
}
```

### ‚ùå Mistake 2: Blocking Signup on Verification Errors

```javascript
// BAD: User can't sign up if verification service is down
try {
  const result = await verifio.verify(email);
  if (result.state !== 'deliverable') {
    throw new Error('Invalid email');
  }
} catch (error) {
  return res.status(400).json({ error: 'Cannot verify email' });
}
```

```javascript
// GOOD: Graceful degradation
try {
  const result = await verifio.verify(email);
  if (result.state === 'undeliverable') {
    return res.status(400).json({ error: 'Invalid email' });
  }
} catch (error) {
  // Log error but allow signup with manual review flag
  console.error('Verification failed:', error);
  await createUser({ email, needsManualReview: true });
}
```

### ‚ùå Mistake 3: Ignoring the Score

```javascript
// BAD: Only checking state
if (result.state === 'deliverable') {
  // Proceed
}
```

```javascript
// GOOD: Use score for better decisions
if (result.state === 'deliverable' && result.score >= 70) {
  // High quality email
} else if (result.state === 'deliverable' && result.score < 70) {
  // Valid but low quality - maybe flag for review
}
```

### ‚ùå Mistake 4: Not Using Request IDs for Debugging

```javascript
// BAD: No way to debug issues
catch (error) {
  console.error('Verification failed');
}
```

```javascript
// GOOD: Log request ID for support
catch (error) {
  console.error('Verification failed:', {
    error: error.message,
```

## Troubleshooting

### Issue: "Invalid API key" error

**Possible causes:**
- API key is incorrect or has typos
- Using wrong header format (should be `Bearer YOUR_KEY`)
- API key has been revoked or expired
- Using test key in production or vice versa

**Solution:**
```bash
# Verify your request format
curl -X POST "https://verifio.email/api/verify/v1/verify" \
  -H "Authorization: Bearer vrf_sk_live_YOUR_KEY_HERE" \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com"}'
```

### Issue: Verification times out frequently

**Possible causes:**
- Target domain's SMTP server is very slow
- Network connectivity issues
- Domain's mail server is unresponsive

**Solution:**
- Implement retry logic with exponential backoff
- Consider marking as `unknown` after 2-3 timeouts
- Check if specific domains consistently timeout

### Issue: Running out of credits unexpectedly

**Possible causes:**
- Not monitoring credit usage
- Verifying same emails multiple times
- Infinite loops in code

**Solution:**
```javascript
// Implement credit monitoring
const CREDIT_THRESHOLD = 100;

async function checkCreditsAndAlert() {
  const credits = await verifio.getCredits();
  if (credits < CREDIT_THRESHOLD) {
    await sendAlert(`Low credits: ${credits} remaining`);
  }
}

// Run periodically
setInterval(checkCreditsAndAlert, 3600000); // Every hour
```
